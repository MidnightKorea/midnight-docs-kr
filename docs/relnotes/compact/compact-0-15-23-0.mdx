## Compact compiler 0.23.0 (Compact language 0.15.0) release notes

Compact 컴파일러 버전 0.23.0을 릴리스합니다. Compact 언어 버전이 0.14.0에서 0.15.0으로 올라갑니다. 이 컴파일러는 곧 예정된 Midnight [테스트넷 업그레이드](https://midnight.network/blog/upcoming-testnet-02-upgrade-all-you-need-to-know)에 포함됩니다.

테스트넷 업그레이드는 **호환성을 깨는** 변경입니다. 업그레이드된 테스트넷에서 컨트랙트를 실행하려면 Compact 컴파일러 0.23.0 이상으로 다시 컴파일해야 합니다. 증명 시스템이 변경되었으므로, 컴파일러가 생성하는 새 증명자/검증자 키가 필요합니다.

:::important

컴파일러는 증명자/검증자 키 생성에 별도 바이너리 `zkir`을 사용합니다. `error: unrecognized subcommand 'compile-many'` 오류가 나타나면 `zkir` 바이너리 버전이 호환되지 않는 것입니다. PATH에서 찾는 `zkir`이 컴파일러 0.23.0에 포함된 것인지 확인하세요.

:::

언어 변경 사항도 포함되어 있어 Compact 언어 버전이 0.15로 올라갔습니다.

버그 수정과 컴파일러 개선 사항도 다수 포함되어 있으며, 아래에서 자세히 설명합니다.

## Summary of Changes

언어 변경:

- [호환성을 깨는 변경] 최대 `Field` 값 변경
- [비호환성 변경 아님] 후행 쉼표/세미콜론을 더 많은 곳에서 허용

버그 수정:

- 잘못된 `Field` 산술 오버플로 및 언더플로 수정
- 잘못된 공통 부분식 제거로 인한 크래시 수정
- 원장 ADT 타입으로의 캐스트 시도 시 크래시 수정
- 큰 정수 리터럴 관련 크래시 수정

컴파일러 개선:

- 키 생성 중 진행률 표시기
- 컴파일러가 이제 오래된 ZKIR 파일을 제거합니다
- 개발자가 나눗셈을 사용하려 할 때 더 나은 오류 메시지

## Language Updates

Compact 언어에 몇 가지 변경이 있습니다. 아래 두 기능을 사용하는 컨트랙트라면, 다음과 같이 언어 버전 0.15를 명시하세요:

```compact
pragma language_version 0.15;
```

### [Breaking] The maximum `Field` value has changed

Midnight 테스트넷 업그레이드에서 ZK 증명 시스템이 변경됩니다. zkSNARK 생성에 사용하는 타원 곡선이 Pluto-Eris에서 BLS12-381로 전환됩니다. 이에 따라 DApp TypeScript 코드의 `MAX_FIELD` 최대 필드 값이 달라집니다.

**호환성을 깨는** 변경입니다. 이 값에 의존하는 컨트랙트는 수정이 필요할 수 있습니다.

### [Non-breaking] Trailing commas and semicolons are now allowed in more places

TypeScript는 쉼표로 구분된 목록에서 후행 쉼표를 허용하지만, Compact에서는 모든 위치에서 허용하지 않았습니다. 이번에 대부분의 구문 항목 목록에 후행 쉼표 지원을 추가했습니다. 단, 시퀀스 표현식에서는 허용하지 않습니다(TypeScript와 동일).

아직 회로/위트니스 호출의 인수 목록과 `map`/`fold` 표현식에서는 후행 쉼표가 동작하지 않습니다. 이 버그는 향후 릴리스에서 수정 예정입니다.

다음은 후행 쉼표를 보여주는 예시입니다:

```compact
circuit foo<T, S,>(x: T, y: S,): [S, T,] {
  return [y, x,];
}

export circuit bar(): Uint<8> {
  foo<Uint<8>, Uint<8>,>(1, 2);
  foo<Uint<8>, Uint<8>>(1, 2,);  // 의도하지 않은 구문 오류 (컴파일러 버그)
  return 1, 2, 3,;  // 의도된 구문 오류 (TypeScript와 일치)
}
```

**비호환성 변경이 아닙니다.** 기존에 컴파일/실행되던 프로그램은 동일한 결과로 계속 동작합니다.

## Bug Fixes

### Fix for incorrect `Field` arithmetic overflow and underflow

Compact의 `Field` 타입은 암호화 백엔드에 따라 최대값이 고정됩니다. `Field` 덧셈/곱셈의 오버플로는 래핑(최대 필드 값 + 1의 나머지)되어야 하고, 뺄셈 언더플로도 마찬가지입니다. 그러나 컴파일러가 생성한 JavaScript 코드에서 이 래핑이 올바르게 구현되지 않았습니다. 잘못된 결과뿐 아니라, 증명은 올바른 산술을 사용하므로 트랜잭션 제출이 실패하는 경우도 있었습니다. 이제 [언어 레퍼런스](https://docs.midnight.network/develop/reference/compact/compact-reference#binary-arithmetic-expressions)대로 올바르게 동작합니다.

### Fix for a crash due to incorrect common subexpression elimination

Compact 컴파일러는 증명의 크기와 복잡성을 줄이기 위해 여러 회로 최적화를 수행합니다. 그중 "공통 부분식 제거"에서, 부분식의 첫 번째 발생이 데드 코드 제거로 이미 삭제된 코드에 있을 때 올바르게 처리되지 않는 문제가 있었습니다. 다음 예시를 보세요:

```compact
export circuit foo(x: Field): [] {
  if (false) {
    assert (x != 0) 'oops1';
  }
  assert (x != 0) 'oops2';
}
```

컴파일러가 `x != 0`의 두 번째 발생을 제거하고 첫 번째 것을 참조하려 했지만, 첫 번째 발생도 도달 불가능하여 이미 제거된 상태였습니다. 이로 인해 내부 단언 실패로 크래시가 발생했으며, 이 문제를 수정했습니다.

### Fix for a crash when trying to cast to a ledger ADT type

`Counter`나 `Map` 같은 원장 ADT 타입은 일반 Compact 타입으로 사용할 수 없습니다. 값을 원장 ADT 타입으로 캐스트하면 오류 메시지를 표시해야 하지만, 기존에는 내부 오류로 크래시했습니다. 이제 "invalid context for a ledger ADT type" 오류를 올바르게 표시합니다.

### Fix for crashes involving large integer literals

Compact 정수는 암호화 백엔드에 따른 최대값으로 제한됩니다. 이 값을 초과하는 정수 리터럴이 있으면, 경우에 따라 오류를 보고하거나 크래시했습니다.

크래시하던 경우를 수정하여 컴파일 타임 오류를 표시하도록 변경했습니다.

## Compiler Improvements

### There is a progress meter during key generation

컴파일러는 zkSNARK용 암호화 키 쌍(증명자/검증자 키)을 생성합니다. 내보낸 회로마다 한 쌍씩 만들어지며, 키 생성은 보통 컴파일에서 가장 느린 단계입니다. 일부 사용자는 이 과정에서 컴파일러가 멈춘 것으로 오해하기도 했습니다.

키 생성 중 진행률 표시기를 추가하여, 진행 상황과 남은 작업량을 확인할 수 있도록 했습니다.

### The compiler removes stale ZKIR files

컴파일러는 출력 디렉터리의 `zkir` 하위 디렉터리에 내보낸 회로별로 ZKIR 파일을 생성합니다. 이 파일은 증명자/검증자 키 생성에 사용됩니다. 기존에는 컴파일러가 오래된 ZKIR 파일을 남겨두어 불필요한 키 생성이 일어났고, 증명 시스템이 업데이트된 경우 키 생성이 실패할 수도 있었습니다.

예를 들어, `foo`와 `bar` 회로를 내보내는 컨트랙트에서 `bar`를 제거해도, `bar`의 ZKIR 파일이 남아 있어 컴파일러가 계속 `bar`의 키를 생성하려 했습니다.

이제 컴파일러가 실행 시 오래된 ZKIR 파일을 자동으로 제거합니다. 키 생성이 컴파일에서 가장 느린 부분이므로 체감 성능이 향상됩니다.

### Better error message when a developer attempts to use division

Compact은 나눗셈 연산자(`/`)를 아직 지원하지 않습니다. 개발자가 `/`를 사용하면, 컴파일러가 이를 주석 시작으로 인식하려 해서 "unexpected character ' '"라는 엉뚱한 오류를 표시했습니다.

이제 오류 메시지를 개선하여, `+`, `-`, `*` 같은 이항 연산자가 필요한 위치라고 안내합니다. 개발자가 `/`가 지원되지 않음을 바로 알 수 있습니다.
