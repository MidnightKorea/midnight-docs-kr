## Version

- **Version:** Compact toolchain 0.28.0, Compact language 0.20.0
- **Date:** 2026-01-28
- **Environment:** Preview, Preprod

## High-level summary

Compact 툴체인 버전 0.28.0을 릴리스합니다.
이 버전은 Compact 언어 0.20.0을 지원합니다.

:::important

이 버전은 Midnight 네트워크의 새 환경인 "Preproduction Testnet"(Preprod)과 "Preview Testnet"(Preview)을 대상으로 하는 첫 번째 툴체인입니다.
컴파일러 0.28.0 이상으로 컴파일한 컨트랙트는 이전 환경("testnet", "testnet-02")에서 동작하지 않습니다.
반대로, 0.28.0 이전 버전으로 컴파일한 컨트랙트는 Preprod/Preview 환경에서 동작하지 않습니다.

:::

Midnight 네트워크의 새 Preprod/Preview 환경 출시에 맞춘 중요한 릴리스입니다.
호환성을 깨는 변경이 포함되어 있으므로 릴리스 노트를 꼼꼼히 확인해 주세요.

## Audience

이 릴리스 노트는 Compact 컨트랙트 개발자와 Compact 런타임을 사용하는 DApp 개발자를 대상으로 합니다.

## Summary of updates

Preprod/Preview 환경은 Midnight 원장 버전 7을 사용합니다(기존 테스트넷은 원장 버전 4). 이 원장 버전에서 네이티브 비차폐(unshielded) 토큰을 새로 지원합니다.

- Compact 표준 라이브러리에 비차폐 토큰용 새 API가 추가되었으며, 기존 API 일부의 이름이 변경되었습니다(호환성을 깨는 변경).
- 구조적("투명") 및 명목적("불투명") 타입 별칭 정의를 모두 지원합니다.
- 선택적 모듈 가져오기와 이름 변경을 지원합니다.
- `Uint` 범위의 의미가 변경되었습니다.
- 최대 표현 가능한 부호 없는 정수가 더 작아졌습니다.
- 다양한 크기에 대한 새로운 컴파일러 강제 한계가 있습니다.
  기존에 컴파일되던 프로그램이 컴파일되지 않을 수 있으므로 호환성을 깨는 변경입니다.
  단, 해당 컨트랙트는 실제로 동작할 가능성이 낮았습니다.
- 표준 라이브러리 타입 `CurvePoint`가 `NativePoint`로 이름이 변경되었습니다. 내보내지지 않은 내부 타입의 명목적 타입 별칭입니다.
- 표준 라이브러리가 `NativePoint`의 접근자로 새 회로 `NativePointX`와 `NativePointY`를 제공합니다.
- 생성된 JavaScript 코드와 Compact 런타임이 CJS 대신 ESM(ECMAScript 모듈)을 사용합니다.
  DApp에 대한 호환성을 깨는 변경이지만, ESM만 사용하는 개발자에게는 오히려 간편해집니다.
- 상향 캐스트가 튜플 참조/슬라이스의 상수 인덱스 인식을 방해하지 않도록 수정했습니다.
- 범위 밖 바이트 값 인덱스를 컴파일러가 더 일찍 감지합니다. 기존에 오류 없이 컴파일되던 프로그램에서 추가 오류가 표시될 수 있습니다.
- TypeScript와 JavaScript의 예약어가 이제 Compact의 예약어에 포함됩니다.
- `for` 루프 본문에서 `return`이 더 이상 허용되지 않습니다.
- 컴파일러가 더 이상 필요하지 않은 특정 회로에 대해 ZKIR 코드와 증명자/검증자 키를 생성하지 않습니다.
- 회로 시그니처, 호출, 익명 회로의 포매팅이 개선되었습니다.
- `if`/`else if` 문의 포매팅이 개선되었습니다.
- 포매터의 줄 길이를 설정할 수 있으며, 기본값은 100컬럼입니다.
- 여러 버그 수정과 성능 개선이 포함되어 있습니다.

## New features

이 릴리스에 추가된 새 기능의 상세 내역입니다.

### Unshielded tokens

비차폐 토큰을 다루기 위한 새로운 표준 라이브러리 회로와 원장 커널 연산이 있습니다.

비차폐 토큰의 발행과 전송을 위한 새 회로: `mintUnshieldedToken`, `sendUnshielded`, `receiveUnshielded`.
비차폐 토큰 잔액을 다루기 위한 새 회로: `unshieldedBalance`, `unshieldedBalanceLt`, `unshieldedBalanceGt`, `unshieldedBalanceLte`, `unshieldedBalanceGte`.

원장 커널에 새 연산: `mintUnshielded`, `claimUnshieldedCoinSpend`, `incUnshieldedOutputs`, `incUnshieldedInputs`.

비호환성 변경은 아니지만, 아래에 설명하는 호환성을 깨는 이름 변경이 수반됩니다.

### Type aliases

구조적("투명") 및 명목적("불투명") 타입 별칭을 모두 지원합니다.

구조적 타입 별칭은 TypeScript와 동일합니다. `type Name = Type`으로 선언하면 `Name`이 `Type`의 별칭이 됩니다. 두 타입은 완전히 동일합니다.

예를 들어 `type U32 = Uint<32>`는 `U32`를 `Uint<32>`와 동등하고 교환 가능하게 정의합니다.

명목적 타입 별칭은 기본 타입의 표현을 가진 새로운 별개 타입을 만듭니다. TypeScript에는 없는 기능입니다. `new type Name = Type`으로 선언하면, `Type`과 동일한 값을 가지되 별개의 타입으로 취급되는 `Name`이 만들어집니다. `Name`은 `Type`의 서브타입도 슈퍼타입도 아니며, 다음과 같은 방식으로 호환됩니다:

- `Name` 값을 `Type`에 허용되는 기본 연산(산술 등)에 사용할 수 있습니다.
- `Name`과 `Type` 사이에 명시적 캐스트가 가능합니다.

예를 들어, `type RGB = Vector<3, Uint<8>>`로 정의하면 `RGB`는 별개 타입이 됩니다. `RGB` 값은 `Vector<3, Uint<8>>`처럼 참조/슬라이스가 가능하지만, `Vector<3, Uint<8>>`을 기대하는 함수에는 명시적 캐스트 없이 전달할 수 없습니다.

산술 연산에 명목적 타입 별칭 `T` 값이 들어오면, 다른 피연산자도 `T` 타입이어야 하고 결과도 `T`로 캐스트됩니다. 결과를 `T`로 표현할 수 없으면 타입 오류가 발생합니다.

두 종류의 별칭 모두 제네릭 매개변수를 가질 수 있습니다. 예를 들어, `type V3<T> = Vector<3, T>` 또는 `new type VField<#N> = Vector<N, Field>`.

### Selective module importing and renaming

Compact 모듈에서 개별 요소만 선택적으로 가져오거나, 가져온 요소의 이름을 변경할 수 있습니다.

TypeScript/JavaScript의 ES 모듈과 동일한 방식입니다.

예를 들어:

```compact
import { getMatch, putMatch as originalPutMatch } from Matching;
```

`getMatch`는 그대로, `putMatch`는 `originalPutMatch`라는 이름으로 가져오며, `Matching`의 다른 요소는 가져오지 않습니다. 이름 변경으로 충돌을 피할 수 있고, 필요한 것만 가져와 모듈에 새 요소가 추가될 때의 충돌도 방지합니다.

선택적 가져오기와 이름 변경은 Compact의 기존 모듈 접두사와 함께 동작합니다:

```compact
import { getMatch, putMatch as originalPutMatch } from Matching prefix M_;
```

이렇게 하면 `getMatch`를 `M_getMatch`로, `putMatch`를 `M_originalPutMatch`로 가져옵니다.

`CompactStandardLibrary`가 모듈이므로 표준 라이브러리에서도 선택적 가져오기와 이름 변경이 동작합니다.

기존 import 형식도 계속 지원됩니다. 다만 `from`이 예약어로 추가되었으므로 기술적으로는 호환성을 깨는 변경입니다.

## Improvements

기존 기능의 개선 사항입니다.

### There are improvements to the formatter

- 여러 줄에 걸친 회로 시그니처, 호출, 익명 회로의 포매팅을 변경했습니다. 기존에는 수직 정렬이 과도하여 코드가 오른쪽으로 밀려나는 경향이 있었는데, 수평 공백을 크게 줄였습니다.
- `if`/`else if` 연쇄 문의 포매팅이 C 스타일 언어(TypeScript, JavaScript 등)의 표준 관행과 맞지 않던 문제를 개선했습니다.
- 포매터 줄 길이를 설정할 수 있습니다. `compact format --line-length N`으로 컬럼 수를 지정하며, 기본값은 100입니다.

### Upward casts do not prevent tuple constant indexing

이전 릴리스에서 숫자 리터럴이 아닌 표현식으로도 튜플, 벡터, 바이트 값을 인덱싱할 수 있도록 했습니다. 단, 컴파일러가 해당 표현식의 상수 값을 결정할 수 있어야 합니다.

상수값 분석이 튜플 값의 상향 캐스트(슈퍼타입으로의 캐스트)를 올바르게 인식하지 못하던 문제를 수정했습니다.

이에 따라 상수 인덱싱을 사용하는 더 많은 프로그램이 타입 체크와 컴파일을 통과합니다.

### The compiler no longer generates ZKIR code and cryptographic keys for some circuits that do not need it

컴파일러는 Midnight 원장을 사용하는 회로에만 ZKIR 코드와 증명자/검증자 키가 필요합니다. 기존에는 순수 회로(원장 미사용 + 위트니스 미호출)에 대해서는 생성하지 않았지만, 위트니스 호출 때문에만 비순수인 회로(원장은 미사용)에 대해서는 불필요하게 생성했습니다.

이런 회로에 대한 ZKIR/키 생성을 제거했습니다. 키 생성이 컴파일에서 가장 느린 부분이므로, 컴파일 속도가 향상됩니다.

## Breaking changes

컨트랙트 수정 등 조치가 필요할 수 있는 호환성을 깨는 변경입니다.

### Name changes to APIs for shielded tokens

차폐(shielded) 토큰 관련 표준 라이브러리 회로, 원장 커널 연산, 구조체의 이름이 변경되었습니다. 온체인 런타임 명명과 일치시키고 비차폐 API와의 혼동을 방지하기 위함입니다.

**동기:** 이름에 `Shielded`를 포함하여 비차폐 대응물과 명확히 구별합니다.

호환성을 깨는 변경이며, `compact fixup` 도구로 자동 수정할 수 있습니다. Compact devtools 0.3.0 이상이 필요합니다. `compact self update`로 최신 버전으로 업데이트하세요.

fixup 도구 사용 방법:
1. 컨트랙트의 `language_version` pragma를 0.19 이상, `compiler_version` pragma를 0.27 이상으로 변경하세요. 그렇지 않으면 도구가 변경을 적용하지 않습니다.
2. `.compact` 파일에 `compact fixup <file>`을 실행하세요. 수정된 컨트랙트가 표준 출력으로 나옵니다.

fixup 도구는 이름 변경 등으로 포매팅이 깨질 수 있어, 항상 Compact 포매터로 재포맷합니다. 다른 변경이 없더라도 재포맷은 수행됩니다.

### The meaning of `Uint` ranges has changed

범위 제한 부호 없는 정수(`Uint<0..n>` 형태, `Uint<n>` 형태가 **아닌**)의 범위가 변경되었습니다. 왼쪽은 포함(inclusive, 항상 0), 오른쪽은 제외(exclusive, `n`)입니다. 기존에는 양쪽 모두 포함이었습니다.

예를 들어, `Uint<0..3>`은 이제 {0, 1, 2}를 의미합니다. 기존에는 {0, 1, 2, 3}이었습니다.

**동기:** `for` 루프의 범위 의미와 일관되지 않아 미묘한 버그를 유발할 수 있었기 때문입니다.

이것은 범위 제한 부호 없는 정수에 대한 호환성을 깨는 변경입니다. 더 일반적으로 사용되는 크기 지정 부호 없는 정수(`Uint<n>`)는 변경되지 않았습니다.

`compact fixup` 도구는 기본적으로 이를 업데이트하지 않지만, `--update-Uint-ranges` 플래그를 전달하면 수정을 시도합니다:

- `n`이 숫자 리터럴인 `Uint<n>`의 경우, `n`+1과 같은 숫자 리터럴 `m`으로 `Uint<m>`로 변경합니다.
- `n`이 제네릭 크기 매개변수인 `Uint<n>`의 경우, 경고를 발생시킵니다.

제네릭 크기 매개변수를 사용하는 코드를 수정하려면, 제네릭의 사용처(숫자 리터럴로 인스턴스화되는 곳)를 찾아 업데이트해야 합니다.

### The maximum representable unsigned integer value has changed to be smaller

범위 제한 부호 없는 정수의 온체인 표현은 `Field`의 전체 바이트 수(현재 31)로 제한됩니다. 이 범위를 벗어나는 값을 사용하면 원인을 알 수 없는 증명 실패가 발생할 수 있었습니다.

따라서 `Uint<n>`의 `n`의 최대값과 `Uint<0..m>`의 `m`의 최대값이 다음으로 줄었습니다:
`n`은 248(8 * 31), `m`은 452312848583266388373324160190187140051835877600158453279131187530910662656(2^248).

**동기:** 증명 서버가 이보다 큰 부호 없는 정수를 올바르게 처리하지 못합니다.
이것은 `Uint` 타입에 대한 호환성을 깨는 변경입니다.
최대 `Field` 값은 변경되지 않았습니다.

### There are compiler-enforced size bounds

컴파일러가 이제 튜플, 벡터(`Vector`), 바이트 값(`Bytes`) 타입에 대해 최대 요소 크기 한계를 부과합니다. 이 최대 요소 크기 한계는 2^24(= 16,777,216)입니다.

또한 `MerkleTree`와 `HistoricMerkleTree` 원장 ADT 타입에 대한 최대 깊이 한계가 있습니다. 이 최대 한계는 깊이 32입니다.

**동기:** 이 한계를 초과하는 프로그램은 실질적으로 동작하지 않았습니다. 튜플/벡터/바이트 값의 경우 키 생성이 실패하거나 컴파일러가 종료되지 않았고, Merkle 트리 깊이 제한은 Midnight 원장 자체의 제한에 따른 것입니다.

### The standard library type `CurvePoint` is renamed to `NativePoint`

표준 라이브러리 타입 `CurvePoint`가 `NativePoint`로 변경되었습니다. `NativePoint`는 내보내지지 않은 내부 타입의 명목적(불투명) 타입 별칭입니다.

`NativePoint`는 한 쌍의 `Field` 값(아핀 x, y 좌표)을 가진 `struct` 타입으로, `NativePoint { 0, 1 }` 구문으로 생성합니다.

**주의: `NativePoint`의 구조체 표현에 의존하지 마세요.** 다음 릴리스에서 표현이 변경될 수 있습니다.

표준 라이브러리에 `NativePoint` 접근자 회로 `NativePointX`와 `NativePointY`를 도입했습니다. 이 접근자 회로로 아핀 x 및 y 좌표에 접근할 수 있습니다.

코드를 수정하려면 `CurvePoint`를 `NativePoint`로 바꿔야 합니다. 또한 `pt`가 이제 `NativePoint` 타입인 경우 `pt.x` 형태의 표현식을 `NativePointX(pt)`로, `pt.y`도 마찬가지로 바꿔야 합니다.

**동기:** 다음을 준비하기 위해 이 변경을 했습니다:

- `NativePoint`의 표현을 회로에서 더 효율적으로 변경하기 위해 표현을 숨기고자 합니다.
- 다른(외부) 곡선의 포인트를 도입하기 위해, `CurvePoint`라는 이름이 너무 일반적입니다.

또한 접근자 `NativePointX`와 `NativePointY`는 TypeScript 명명 규칙을 따르지 않습니다(소문자로 시작해야 합니다). 향후 릴리스, 가능하면 다음 릴리스에서 이름을 변경할 예정입니다.

### The reserved words from TypeScript and JavaScript are now included as Compact reserved words

TypeScript와 JavaScript의 전체 예약어 목록을 Compact의 예약어로 포함시켰습니다. 일부는 미래 예약어로 간주됩니다. Compact에서 아직 사용되지 않지만 향후 사용을 위해 예약합니다.

**동기:** 새 키워드를 추가할 때마다 호환성을 깨야 하는 상황을 방지하기 위함입니다. Compact 1.0 릴리스 전에 미리 예약하여, 향후 호환성을 깨는 변경을 최소화합니다.

Compact 0.20 예약어의 전체 목록은 [파서 구현](https://github.com/LFDT-Minokawa/compact/blob/v0.28.0/compiler/parser.ss#L56)에서 확인할 수 있습니다. 이 단어 중 하나를 사용하여 컴파일러 오류가 발생하면, 해당 식별자의 이름을 변경해야 합니다.

### Out-of-range constant bytes value indexes are detected earlier in the compiler

컴파일러는 튜플, 벡터, 바이트 값 인덱스가 범위를 벗어나면 오류를 표시합니다(컨트랙트를 컴파일하지 않음). 그러나 바이트 값의 경우, 이 오류 감지가 이전보다 컴파일러에서 더 일찍 이동되었습니다.

호환성을 깨는 변경입니다. 기존에는 오류 감지 전에 코드가 폐기되어 넘어갔지만, 이제 컴파일러 오류가 발생합니다.

**동기:** 새로운 타입 별칭 기능 지원을 위한 내부 구현 변경입니다.

해당 오류를 수정하세요. 기존에 컴파일러가 놓치고 있던 실제 오류입니다.

### Generated JavaScript code and the Compact runtime now uses ES modules (ESM)

컴파일러가 생성하는 JavaScript 코드와 Compact 런타임이 ESM만 사용합니다. 기존에는 CJS(Common JS) 모듈이었습니다.

**동기:** ESM이 공식 표준 JavaScript 모듈 시스템이며, ESM 프로젝트에 CJS를 혼합하면 설정이 복잡해지기 때문입니다.

DApp에 대한 호환성을 깨는 변경입니다. 컨트랙트 코드와 Compact 런타임의 import 방식을 변경해야 합니다.

## Fixed defect list

다수의 버그를 수정했으며, 일부 케이스에서 증명용 회로 표현의 크기가 줄었습니다.

아래 수정 중 일부는 호환성을 깨는 변경입니다:

- 기존에 일부 도달 불가능한 문이 보고되지 않았는데, 이제 올바르게 보고됩니다. 기존에 컴파일되던 컨트랙트가 실패할 수 있으니, 도달 불가능한 코드를 제거하거나 주석 처리하세요.

- 기존에 타입 체크하지 않던 일부 미사용 모듈을 이제 타입 체크합니다. 이전에 발견되지 않던 타입 오류로 컴파일이 실패할 수 있습니다. 타입 오류를 수정하세요.

- 단일 문 컨텍스트(`if` 분기, `for` 루프 본문 등)에서 다중 바인딩/구조 분해 `const` 문을 거부합니다. 상수가 해당 스코프 밖에서 사용될 수 없으므로 프로그래머의 의도와 다를 가능성이 높습니다.

- `Bytes<0>`에서 `Field`/`Uint`로의 양방향 캐스트를 금지합니다. 가끔 동작했지만, 엉뚱한 컴파일러 오류를 유발할 수 있었습니다.

- `for` 루프 본문에서 `return` 문을 명시적으로 거부합니다. 이 구문은 올바르게 동작한 적이 없으며, 기존에는 관련 없는 암호화 오류나 런타임 오동작으로 이어졌습니다.
