# Bulletin board contract

counter DApp과 마찬가지로, [`example-bboard`](https://github.com/midnightntwrk/example-bboard) 저장소에도 컨트랙트와 사용자 상호작용 코드가 별도의 하위 디렉터리로 분리되어 있습니다.
다만 단순한 counter DApp과 달리, 게시판 DApp은 사용자 인터페이스 코드와 UI에 독립적인 핵심 애플리케이션 로직을 추가로 분리합니다. 이러한 구조 덕분에 이번 튜토리얼에서는 명령줄 버전의 게시판을, 이후 파트에서는 웹 기반 버전을 만들 수 있습니다.
- `contract` - 게시판 컨트랙트의 Compact 소스 코드와 관련 TypeScript를 포함합니다
- `api` - 메시지 게시 및 삭제와 같은 DApp의 핵심 동작을 구현하는 TypeScript 소스를 포함하며, `contract` 코드에 의존합니다
- `bboard-cli` - 텍스트 기반 DApp의 명령줄 인터페이스를 포함하며, `contract`와 `api` 코드 모두에 의존합니다.


:::tip

이 페이지는 게시판 컨트랙트의 설계를 단계별로 안내합니다. 완성된 컨트랙트는 페이지 끝에 나타납니다. Midnight에서는
[Compact용 Visual Studio Code 확장](../../compact/compilation-and-tooling/vscode-plugin.mdx)을 사용하여 컨트랙트를 보고 편집하는 것을 권장하며,
이를 통해 구문 강조와 이름 자동 완성 기능을 사용할 수 있습니다.

:::

게시판 시나리오를 위한 Midnight 컨트랙트를 작성하려면 다음 세 가지를 파악해야 합니다:
- 컨트랙트의 공개 상태를 구성하는 요소
- 컨트랙트에서 수행 가능한 공개 작업
- 공개 작업에서 증명 가능한 방식으로 사용되지만 외부에 공유되지 않는 비공개 데이터 및 작업

컨트랙트의 공개 상태와 공개 작업의 트랜잭션 이력은 Midnight 블록체인 ledger에 기록되며, 누구나 이를 검증할 수 있습니다. 비공개 데이터는 DApp 사용자의 로컬 환경 밖으로 전송되지 않습니다.

## Public ledger state on the blockchain

게시판의 공개 ledger 상태는 네 개의 값으로 구성됩니다:
1. 상태: 비어있음(vacant) 또는 사용중(occupied)
2. 메시지
3. 현재 어떤 특정 게시물이 현재인지 식별하는 카운터
4. 현재 게시물을 작성한 사용자가 생성한 공개 토큰으로, 사용자의 비공개 신원을 유추할 수 없습니다.

세 번째 값은 다른 값들보다 직관적이지 않을 수 있습니다. 이전 페이지에서 언급한 '해당 메시지' 제약에 해당하는 값으로, 예를 들어 값이 15이면 '현재 메시지는 15번째 게시물'이라는 의미입니다. 소프트웨어 잠금을 구현해 본 경험이 있다면, 게시판이 사용중이 되는 시점이 아니라 비어있게 되는 시점에 이 카운터를 증가시키는 것이 올바르다는 점을 쉽게 이해할 수 있을 것입니다. 게시판이 비워지는 시점이 이전 번호의 게시물이 더 이상 존재하지 않게 되는 시점이기 때문입니다.

네 번째 값은 소유자의 신원과 게시 순서 번호(카운터)의 비가역적 해시여야 합니다. 누구도 토큰으로부터 메시지를 게시한 사람을 알아낼 수 없지만, 게시물을 작성한 사용자는 신원 검증 의무를 충족하기 위해 해당 토큰을 안정적으로 다시 도출할 수 있습니다.

이 ledger 선언에서 사용된 타입에 대해 몇 가지 참고할 점이 있습니다:
- Compact에서는 게시판의 비어있음/사용중 상태를 표현하는 열거형 타입처럼 새로운 타입을 선언할 수 있습니다. 선언에 `export` 수정자를 추가하면 Compact 컴파일러가 열거형 타입과 그 값에 대한 TypeScript 표현을 자동으로 생성합니다.
- `STATE`, `Maybe`, `Bytes` 같은 Compact 타입을 가진 ledger 필드는 circuit를 통해 값을 업데이트할 수 있는 가변 셀입니다.
- `Counter` 타입은 0으로 자동 초기화되며, circuit를 통해 증가시킬 수 있습니다.
- Compact의 내장 타입인 `Opaque`는 컨트랙트에서 내부 구조가 중요하지 않은 값을 나타냅니다.
- 표준 라이브러리의 `Maybe` 타입은 값이 있을 수도 없을 수도 있는 선택적 값을 표현합니다. `some` 또는 `none` 생성자로 값을 만듭니다.
- 소유자의 신원 토큰은 256비트(32바이트) 해시입니다.
- 표준 라이브러리 `CompactStandardLibrary`에는 `Maybe`, `some`, `none`, `Counter`가 포함되어 있습니다. (`Opaque`와 `Bytes`는 Compact 내장 타입입니다.)

ledger가 생성될 때 컨트랙트가 `owner` 필드를 명시적으로 초기화할 필요는 없습니다. 그 값이 아직 의미가 없기 때문입니다.
생성자에서 명시적으로 초기화되지 않은 모든 ledger 필드는 해당 타입에 기본값이 있는 경우 기본값으로 초기화됩니다.

## Enforcing the contract: circuits

게시판을 사용하는 모든 사람은 규칙을 준수해야 하며, 스스로 컨트랙트의 조건을 충족했음을 증명해야 합니다. 게시판을 변경할 때는 규칙을 따랐다는 proof를 제출하고, 관찰자는 이 proof를 빠르게 검증할 수 있습니다. proof를 생성하는 데 사용된 원본 데이터에 접근하지 않고도 검증할 수 있는 이러한 proof를 *영지식 증명(zero-knowledge proof)*이라 하며, *수학적 circuit*를 사용하여 구현합니다.

이 개념 자체는 새로운 것이 아닙니다. 영지식 증명에 관한 주요 논문 중 일부는 이미 1980년대에 발표되었습니다. 최근의 발전은 이러한 proof를 사람의 개입 없이 자동으로 생성하고 검증하는 방법, 그리고 공개 블록체인과 결합하는 방법에 관한 것입니다.

이 분야에서 *Midnight가 특별히 기여한 점*은 영지식 기반 스마트 컨트랙트와 이를 뒷받침하는 circuit 정의를 *일반 프로그래머도 접근할 수 있도록* 만든 것입니다.

`post` 작업을 Compact `circuit` 정의로 작성하는 것부터 시작합니다. 컨트랙트의 이 부분에서 충족해야 할 주요 의무는 게시판의 첫 번째 규칙입니다: 게시판이 비어 있을 때만 게시할 수 있습니다.

`circuit` 정의는 함수 정의와 매우 유사하며, 입력 매개변수와 반환 값을 지정할 수 있습니다. ledger의 현재 상태는 circuit 정의 내에서 암시적으로 접근할 수 있습니다. `enum STATE` 타입과 마찬가지로 이 정의에도 `export` 수정자가 추가되어 TypeScript에서 `post` circuit를 호출할 수 있습니다.

컨트랙트 의무 사항을 시행하기 위해 `assert` 문을 사용합니다. `assert`는 불리언 표현식이 참인지 확인하며, 거짓이면 트랜잭션이 중단되고 지정된 메시지와 함께 실패를 보고합니다. 위의 `assert`는 메시지 게시 시 게시판이 비어 있는지 확인합니다(규칙 1).

`post` circuit가 ledger의 `owner` 필드에 기록하는 이 생성된 '공개 키'는 무엇입니까? 이는 게시물의 순서 번호와 네트워크를 통해 전송되지 않는 사용자의 비밀 키를 포함하는 문자열을 해싱하여 도출됩니다. 위의 코드는 동일한 컨트랙트에 정의된 '헬퍼' circuit `publicKey`에 대한 호출을 사용했습니다. 다음은 그 정의입니다:

## Accessing private state

그렇다면 `post` circuit에서 필요한 비밀 키를 어떻게 가져올까요? `localSecretKey` 함수는 circuit가 될 수 없습니다. circuit의 반환 값은 공개적으로 검증 가능하기 때문에, 절대 ledger에 노출되어서는 안 되는 이 값을 circuit로 반환할 수 없기 때문입니다.

이런 종류의 함수를 *witness*라고 합니다. Witness 함수는 각 DApp이 로컬에서 관리하는 비공개 상태에 대한 접근을 제공합니다. 컨트랙트에서는 witness의 구현을 정의하지 않고 존재만 선언하며, 실제 구현은 DApp이 담당합니다. 컨트랙트에서의 선언은 다음과 같습니다:

witness의 반환 값은 비공개 데이터로 취급됩니다. Compact 컴파일러는 프로그램 전체에서 이를 추적하여 공개 ledger 상태에 유출되지 않도록 방지합니다. 구체적으로, `post` circuit에서 `localSecretKey()`의 결과는 비공개로 간주됩니다. 이 값이 `publicKey`에 전달되므로 `publicKey`의 결과도 비공개로 간주됩니다(최소한 비공개 데이터에 기반한 것으로 취급됩니다). 이 값을 공개 ledger 상태에 기록하려고 하면 컴파일러가 오류를 발생시킵니다. 하지만 이 경우, 소유자 신원과 순서 카운터의 해시는 소유자의 신원을 유출하지 않습니다. 공개 키 값을 `disclose`로 감싸면 Compact 컴파일러에게 이 공개가 의도적이라는 것을 알리게 됩니다.

`bboard-tutorial`의 `contract` 하위 디렉터리에는 빌드 과정에서 Compact 컴파일러를 자동으로 실행하는 빌드 스크립트가 있지만, 이번에는 학습 목적으로 컴파일러를 직접 실행해 보겠습니다.

[Midnight Compact 컴파일러 실행](../../compact/compilation-and-tooling/compiler-usage.mdx) 가이드에 따라 Compact 컴파일러 디렉터리를 `PATH`에 추가했고, 현재 디렉터리가 `contract` 디렉터리라고 가정하면, 다음 명령으로 `bboard.compact`를 컴파일합니다:

```shell
compact compile src/bboard.compact src/managed/bboard
```

컴파일러를 실행하면 각 공개 circuit(`post`, `takeDown`, `publicKey`)에 대한 circuit 복잡도 메시지가 표시됩니다. 오류 메시지가 나타나면 코드에 실수가 없는지 확인하세요. 도움이 필요하면 Midnight 개발자 관계 팀이나 Discord에서 다른 개발자에게 문의하세요.

`src/managed/bboard/contract/index.d.cts`에서 Compact 컴파일러가 컨트랙트에 대해 생성한 TypeScript API를 볼 수 있습니다. DApp은 이 API를 사용하여 컨트랙트를 배포하고 circuit를 호출합니다.
