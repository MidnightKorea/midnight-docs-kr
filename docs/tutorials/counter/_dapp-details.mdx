# counter DApp

## Project structure

DApp을 여러 서브 프로젝트로 분할하면 유용합니다. 특히 웹 기반 UI를 가진 복잡한 DApp에서는 핵심 로직을 사용자 경험과 독립적으로 테스트할 수 있어야 하기 때문입니다. counter처럼 간단한 DApp도 코드가 두 개의 서브 프로젝트로 나뉘어 있습니다:
- `contract` - 이 서브 프로젝트에는 컨트랙트 자체와 로컬 private 상태의 구현을 정의하는 지원 코드, 그리고 컨트랙트의 로직을 테스트하는 코드가 포함됩니다.
- `counter-cli` - 이 서브 프로젝트는 DApp의 명령줄 인터페이스를 정의합니다. `contract` 서브 프로젝트의 출력에 의존합니다.

각 서브 프로젝트의 최상위 디렉터리에서 프로젝트 정의 및 설정 파일을 찾을 수 있으며, 소스 코드는 `src` 하위 디렉터리에 있습니다.

## The `contract` sub-project

이전 페이지에서 컨트랙트 자체를 이미 살펴보았습니다.
컨트랙트가 매우 간단하므로 추가로 작성해야 할 코드는 많지 않습니다.

### Witness implementation

`contract/src/witnesses.ts` 파일을 살펴보세요. counter 컨트랙트는 private 상태를 정의하지 않으므로 witness 함수를 구현할 필요가 없습니다. 하지만 TypeScript에서 컨트랙트를 표현할 때 모든 컨트랙트가 private 상태 타입으로 매개변수화되므로, *어떤 형태로든* private 상태 객체를 정의해야 합니다. 여기서는 빈 객체 타입을 나타내기 위해 `Record<string, never>` 타입을 사용합니다.
따라서 counter의 `witnesses.ts`는 가능한 가장 최소한의 (빈) witness 구현을 제공합니다.

### Generated source

`contract/src/index.ts` 파일은 컨트랙트 서브 프로젝트의 다른 파일에서 정의를 재내보내기합니다. 빈 witness 정의뿐만 아니라 *Compact 컴파일러가 생성한 정의*도 포함됩니다.

이 튜토리얼의 단계를 순서대로 따라왔다면 이미 counter 프로젝트를 빌드했을 것이며, `counter.compact`에 대해 Compact 컴파일러를 실행한 상태일 것입니다.

컨트랙트 서브 프로젝트의 `managed/counter` 하위 디렉터리를 살펴보면 네 개의 디렉터리가 있습니다:
- `contract` - 컨트랙트의 TypeScript/JavaScript API를 정의하는 파일 포함
- `zkir` - 컨트랙트에 정의된 circuit의 중간 표현 포함
- `keys` - 컨트랙트에 정의된 각 circuit의 prover 키와 verifier 키 포함
- `compiler` - 내보낸 circuit의 이름과 JSON 형식의 인수 및 반환 타입을 포함하는 컨트랙트 메타데이터 포함

counter 컨트랙트는 하나의 circuit `increment`만 정의하므로, `keys`에는 `increment`를 위한 prover 키와 verifier 키 쌍 하나가 있습니다. `zkir` 디렉터리에는 `increment.bzkir`과 `increment.zkir` 두 파일이 있습니다. 이들은 각각 `increment` circuit의 ZKIR의 바이너리 표현과 JSON 표현입니다.

생성된 파일 대부분은 직접 검토할 필요가 없지만, `managed/counter/contract/index.d.ts`를 살펴볼 만합니다. `counter.compact` 코드에 직접 대응하는 다양한 타입과 함수가 정의되어 있습니다. 예를 들어:
- 이 컨트랙트의 `Circuits` 타입에는 `increment` 함수가 포함됩니다.
- 이 컨트랙트의 `Ledger` 타입에는 JavaScript `bigint`로 선언된 `round` 필드가 포함됩니다.
- 이 컨트랙트의 `Witnesses` 타입은 컨트랙트가 witness를 선언하지 않으므로 비어 있습니다.

많은 타입이 DApp의 private 상태 구현을 나타내는 타입 `T`에 대해 매개변수화되어 있으며, 컴파일러가 내보낸 circuit 및 ledger 타입에서는 이 타입이 불투명(opaque)합니다.

### Tests

Midnight 튜토리얼의 3부에서 단위 테스트를 다루므로 여기서는 논의하지 않지만, `contract/src/test` 하위 디렉터리의 코드를 자유롭게 살펴볼 수 있습니다. 시뮬레이션 환경에서 컨트랙트를 실행하는 몇 가지 Jest 테스트를 정의합니다.

## The `counter-cli` subproject

counter DApp의 명령줄 인터페이스에 대한 모든 대화형 로직은 `counter-cli/src/index.ts`에 구현되어 있습니다. 이 파일을 검토하기 전에, 전체 DApp이 어떻게 시작되는지 먼저 살펴보겠습니다.

### Different configurations

디렉터리에 네 개의 진입점 파일이 있습니다:
- `testnet-remote.ts` - 로컬 proof 서버를 사용하면서 원격 TestNet의 Midnight 노드 및 Indexer에 연결하는 방식으로, counter DApp을 처음 실행할 때 사용합니다
- `testnet-local.ts` - 자체 TestNet 노드와 indexer를 직접 실행하는 방식으로, 선택적 단계를 시도한 경우에 사용합니다
- `testnet-remote-start-proof-server.ts` - `testnet-remote.ts`와 동일한 설정이지만, proof 서버를 스크립트 내에서 자동으로 시작합니다
- `standalone.ts` - proof 서버, Midnight 노드, Indexer를 모두 TestNet과 독립적으로 로컬에서 실행하며, 단위 테스트용입니다.

### Generated types, instantiated

Compact 컴파일러가 생성한 소스 파일의 많은 타입(예: `Contract`)이 매개변수화되어 있다는 점을 기억하세요. 이러한 타입을 인스턴스화할 때 동일한 타입 인수가 반복되면 코드가 장황해질 수 있습니다. 이를 해결하기 위해 Midnight 팀은 생성된 타입의 인스턴스화된 버전을 정의하는 별도의 파일을 만들어 나머지 코드에서 간결하게 참조할 수 있도록 합니다.

이 패턴의 예를 `common-types.ts` 파일에서 볼 수 있으며, 다음과 같은 `CounterContract` 타입을 정의합니다:

```tsx
export type CounterContract = Contract<CounterPrivateState, Witnesses<CounterPrivateState>>;
```

### DApp

애플리케이션을 구현하는 두 개의 메인 파일은 `counter-cli/src/cli.ts`와 `counter-cli/src/api.ts`입니다. 전자는 애플리케이션의 메인 실행 루프를 포함합니다. 후자는 Midnight 네트워크와 상호작용하기 위한 편의 함수를 포함합니다.

`counter-cli/src/cli.ts`를 이해하려면 끝부분의 `run` 함수부터 시작하세요. 시작 및 오류 처리 코드를 무시하면, DApp의 핵심 동작은 다음과 같습니다:
1. 지갑을 인스턴스화합니다.
2. 컨트랙트와 작업하기 위한 'provider' 컬렉션을 인스턴스화합니다.
3. 메인 사용자 상호작용 루프를 시작합니다.

#### Create a wallet

이 DApp이 웹 브라우저에서 실행되었다면 브라우저의 Lace 지갑 확장에 연결하는 것으로 시작했을 것입니다. 하지만 counter DApp은 브라우저 외부에서 실행되므로 트랜잭션 자금 지원을 위한 헤드리스 지갑을 초기화합니다.

대화형 `buildWallet` 함수에서의 사용자 입력에 따라, DApp은 지갑 시드를 입력받거나 임의의 바이트를 생성하여 시드로 사용합니다. 어느 방식이든 흐름은 `counter-cli/src/api.ts`의 `buildWalletAndWaitForFunds`에 도달합니다.

`WalletBuilder` 클래스는 `buildFromSeed`라는 메인 진입점을 통해 지갑을 생성합니다. 지갑은 indexer를 통해 자금과 트랜잭션 결과를 모니터링하고, 노드를 통해 트랜잭션을 제출하며, proof 서버를 호출하여 트랜잭션 유효성 proof를 생성합니다. 따라서 지갑은 indexer, 노드, proof 서버의 주소를 알아야 합니다.

:::note Wallet SDK v4.0.0 호환성
counter DApp은 다음과 같은 주요 변경 사항이 있는 `@midnight-ntwrk/wallet@4.0.0`을 사용합니다:

- `WalletBuilder.buildFromSeed()`는 **deprecated(지원 중단 예정)**입니다. 아직 작동하지만 향후 버전에서 제거될 예정이므로, 시드 매개변수와 함께 `WalletBuilder.build()`를 사용하세요.
- 지갑 주소와 키는 기본적으로 새로운 **Bech32m 형식**으로 반환됩니다.
- 시드는 더 이상 직렬화된 지갑 상태에 포함되지 않습니다. 지갑을 복원할 때 시드를 명시적으로 다시 전달해야 합니다.
:::


`WalletBuilder.buildFromSeed`에 대한 `getZswapNetworkId` 인수에 주목하세요.
`getZswapNetworkId` 함수는 Midnight.js에서 제공됩니다. 현재 네트워크 ID(`setNetworkId` 호출로 설정된 값)를 가져와 지갑이 이해할 수 있는 형식으로 변환합니다.

지갑이 시작된 후, DApp은 자금이 나타날 때까지 일시 정지합니다. 파일에서 `buildWalletAndWaitForFunds` 바로 위에 있는 `waitForFunds`의 정의를 살펴보는 것이 도움이 될 수 있습니다.
지갑의 상태는 TypeScript에서 observable 스트림으로 나타나므로, `waitForFunds`는 RxJS 함수를 사용하여 지갑 잔액이 0이 아닌 상태를 감시합니다.

지갑의 상태에는 블록체인의 전체 상태와의 동기화 정도에 대한 정보가 포함되어 있습니다.

지갑은 여러 토큰의 잔액을 추적할 수 있으므로, `waitForFunds` 코드에서는 `nativeToken` 잔액, 즉 tDUST를 명시적으로 요청합니다. tDUST 잔액이 0이 아닌 값이 되면 `waitForFunds`가 반환되고, `buildWalletAndWaitForFunds`는 지갑 객체를 반환합니다.

#### Provider

스마트 컨트랙트를 위한 Midnight 프로그래밍 인터페이스는 높은 유연성을 위해 매개변수화가 잘 되어 있습니다. 예를 들어, DApp 개발자가 새로운 종류의 private 상태 저장소를 구현하거나 컨트랙트와 지갑 사이에 잔액 확인 로직을 넣을 수 있습니다. Midnight 라이브러리는 다양한 타입의 *provider*를 필드로 가진 JavaScript 객체를 통해 이러한 기능적 매개변수화를 처리합니다.

대부분의 경우 각 provider 타입에 대한 표준 구현을 그대로 사용하면 되며, Midnight 라이브러리가 이를 제공합니다. 개발자가 지정해야 하는 것은 indexer, proof 서버 등의 위치 정보뿐입니다.

`counterContract` 객체의 `callTx`는 counter 컨트랙트에 대한 호출 트랜잭션을 생성하겠다는 것을 의미합니다. `increment()` 호출은 `increment` circuit에 대한 호출 트랜잭션을 생성하고 제출합니다. 트랜잭션이 제출되고 블록체인에 포함되면 결과가 `finalizedTxData.public`에 저장되며, 여기에는 트랜잭션에 대한 공개 정보가 포함됩니다.


### Viewing Contract State

`providers` 객체의 `publicDataProvider`를 사용하여 블록체인에 대한 정보를 쿼리할 수 있습니다.
counter 컨트랙트에 정의된 `round`의 현재 값을 보려면, `publicDataProvider`를 쿼리하고 컴파일러가 counter 컨트랙트를 위해 생성한 `ledger` 함수를 사용하여 결과를 JavaScript 객체로 변환할 수 있습니다.
